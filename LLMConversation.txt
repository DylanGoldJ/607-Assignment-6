Here’s how you can do it for **“Climate Change”** as a topic:

---

## 1. Use `fq` for topic filtering

```r
library(httr)
library(jsonlite)

url <- "https://api.nytimes.com/svc/search/v2/articlesearch.json"

response <- GET(
  url,
  query = list(
    'fq' = 'subject:("Climate Change")', # filter query
    'api-key' = api_key,
    'page' = 0 # page number, 0-indexed
  )
)

# Parse JSON
json_data <- content(response, as = "parsed", simplifyVector = TRUE)
articles <- json_data$response$docs

# Convert to data frame
df <- as.data.frame(articles)

# Optional: select key columns
df_clean <- df[, c("web_url", "snippet", "lead_paragraph", "source", "pub_date", "section_name")]

head(df_clean)
```

---

### Notes:

Me: subject climate change is giving me null docs

Yeah—that’s a common frustration with the New York Times Article Search API. The issue is that `"Climate Change"` is **not always indexed exactly as a `subject` value**, so your filter:

```r
subject:("Climate Change")
```

often returns empty (`docs = NULL`).

---

## ✅ Why this happens

The NYT API uses a controlled vocabulary, and the actual subject term is usually:

```text
"Global Warming"
```

—not “Climate Change”.

---

## ✅ Fix: use the correct subject term

```r
response <- GET(
  url,
  query = list(
    fq = 'subject:("Global Warming")',
    `api-key` = api_key,
    page = 0
  )
)

json_data <- content(response, as = "parsed", simplifyVector = TRUE)
df <- json_data$response$docs
```

---

## ✅ Better approach: combine query + filter

If you want broader and more reliable results, combine `q` and `fq`:

```r
response <- GET(
  url,
  query = list(
    q = "climate change",
    fq = 'subject:("Global Warming")',
    `api-key` = api_key,
    page = 0
  )
)
```

---

## 🔍 Debug tip (very useful)

Inspect what subjects actually exist in returned articles:

articles <- json_data$response$docs

## ✅ Even more robust option (recommended)

Instead of relying strictly on `subject`, filter after retrieval:

response <- GET(
